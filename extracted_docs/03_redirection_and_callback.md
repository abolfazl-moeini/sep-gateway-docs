# بخش ۳: هدایت خریدار و پاسخ بازگشت تراکنش (صفحات ۹ تا ۱۴)

این بخش شامل راهنمای هدایت خریدار به صفحه پرداخت، ثبت اطلاعات کارت، نمایش رسید به کاربر و دریافت اطلاعات تراکنش پس از بازگشت به سایت پذیرنده (Callback) است.

---

## ۳.۱. هدایت کاربر به صفحه پرداخت (Redirect to IPG)

پذیرنده پس از دریافت توکن پرداخت، خریدار را به همراه توکن با یکی از دو روش زیر به صفحه پرداخت سامان هدایت می‌کند:

### روش اول: هدایت کاربر از طریق متد POST و فرم HTML (توصیه شده)
نمونه فرم HTML جهت هدایت به سمت درگاه پرداخت از طریق متد POST:

```html
<form id="sepForm" action="https://sep.shaparak.ir/OnlinePG/OnlinePG" method="post">
  <!-- توکن دریافتی از مرحله قبل -->
  <input type="hidden" name="Token" value="2c3c1fefac5a48geb9f9be7e445dd9b2" />
  
  <!-- پارامتر GetMethod اختیاری است: مقادیر مجاز: "true" | "false" | "" یا null -->
  <input type="hidden" name="GetMethod" value="true" />
</form>

<script>
  // ارسال خودکار فرم برای هدایت خریدار
  document.getElementById('sepForm').submit();
</script>
```

> [!NOTE]
> **پارامتر GetMethod:** در صورتی که پارامتر `GetMethod` با مقدار `true` ارسال گردد، پس از اتمام فرآیند تراکنش، اطلاعات بازگشتی تراکنش از سمت درگاه به صورت QueryString و با متد **GET** به آدرس بازگشت (RedirectURL) ارسال می‌گردند. در غیر این صورت، این اطلاعات به صورت **POST** فرستاده می‌شوند.

### روش دوم: ساخت لینک پرداخت و هدایت از طریق Redirect (متد GET)
نمونه آدرس هدایت به درگاه پرداخت از طریق متد GET:
```
https://sep.shaparak.ir/OnlinePG/SendToken?token=2c3c1fefac5a48geb9f9be7e445dd9b2
```

> [!WARNING]
> در این روش (روش دوم)، امکان ارسال پارامتر `GetMethod` وجود ندارد و اطلاعات بازگشتی به پذیرنده حتماً با متد POST فرستاده خواهند شد.
> 
> **الزام وجود Referrer:** هدایت کاربر از طریق هر کدام از روش‌های فوق، حتماً بایستی از طریق فرم یا لینکی انجام پذیرد که در سایت پذیرنده قرار دارد تا مرورگر خریدار مقدار هدر `Referrer` را به صورت خودکار و صحیح به درگاه بفرستد؛ در غیر این صورت، امکان ورود به صفحه پرداخت وجود نخواهد داشت.

## ۳.۲. ورود اطلاعات کارت توسط خریدار (Card Details Input)

پس از هدایت خریدار به صفحه پرداخت، خریدار مشخصات کارت خود را در فرم مخصوص درگاه پرداخت الکترونیک سامان وارد می‌کند:
* **شماره کارت (PAN):** شماره ۱۶ رقمی کارت بانکی.
* **رمز دوم کارت (PIN II):** رمز خرید اینترنتی (ایستا یا پویا).
* **تاریخ انقضای کارت (Expire Date):** شامل ماه و سال انقضا.
* **کد امنیتی CVV2:** کد ۳ یا ۴ رقمی پشت کارت.
* **آدرس ایمیل / شماره موبایل:** اختیاری جهت ارسال رسید تراکنش برای خریدار.

> [!NOTE]
> این فرآیند کاملاً در سایت درگاه پرداخت سامان پیاده‌سازی شده و فروشنده نیازی به پیاده‌سازی یا دریافت این اطلاعات در سایت خود ندارد. فروشنده نباید هیچ‌گونه اطلاعات مالی کارت خریدار را دریافت یا ذخیره نماید.

## ۳.۳. نمایش صفحه رسید و بازگشت به سایت پذیرنده (Callback)

پس از انجام تراکنش، خریدار ابتدا صفحه رسید تراکنش را در درگاه سامان مشاهده می‌کند و سپس با کلیک روی دکمه بازگشت (یا به صورت خودکار پس از چند ثانیه)، مرورگر وی اطلاعات پرداخت را به آدرس `RedirectUrl` ثبت‌شده‌ی پذیرنده ارسال می‌نماید. 

پارامترهای ارسالی در پاسخ بازگشت عبارتند از:

| نام پارامتر | نوع داده | توضیحات |
| :--- | :--- | :--- |
| `MID` | String | شماره ترمینال پذیرنده (Terminal ID). |
| `TerminalId` | String | شماره ترمینال پذیرنده (تکرار ترمینال). |
| `State` | String | وضعیت تراکنش به صورت متنی انگلیسی (به عنوان مثال `"OK"` یا `"CanceledByUser"`). |
| `Status` | Integer | وضعیت تراکنش به صورت عددی (به عنوان مثال `2` برای پرداخت موفق). |
| `RRN` | String | شماره مرجع تراکنش در شبکه شتاب. |
| `RefNum` | String | **رسید دیجیتالی تراکنش (یکتا). اگر این فیلد خالی باشد نشان‌دهنده خطا در تراکنش است.** |
| `ResNum` | String | شماره خرید ارسالی توسط پذیرنده در درخواست اولیه. |
| `TraceNo` | String | شماره رهگیری تولید شده توسط سپ. |
| `Amount` | Integer | مبلغ تراکنش به ریال. |
| `Wage` | Integer | مبلغ کارمزد تراکنش (در صورت استفاده از تسهیم). |
| `AffectiveAmount` | Integer | مبلغ کسر شده واقعی از حساب خریدار (در صورت استفاده از سامانه تخفیف). |
| `SecurePan` | String | شماره کارت خریدار به صورت ستاره‌دار/ماسک شده (مثلاً `621986****8080`). |
| `HashedCardNumber` | String | شماره کارت هش شده با الگوریتم **SHA256**. |

## ۳.۴. جدول وضعیت‌های بازگشتی تراکنش (Status Codes Table)

فروشنده می‌تواند بر اساس وضعیت‌های جدول زیر که در فیلدهای `Status` و `State` پاسخ بازگشتی ارسال می‌شوند، موفق یا ناموفق بودن پرداخت در درگاه را متوجه شود:

| کد عددی (`Status`) | کد حروفی (`State`) | توضیحات |
| :---: | :--- | :--- |
| **1** | `CanceledByUser` | کاربر از انجام تراکنش انصراف داده است. |
| **2** | `OK` | پرداخت با موفقیت انجام شد. |
| **3** | `Failed` | پرداخت انجام نشد (خطای عمومی یا خطای بانک صادرکننده). |
| **4** | `SessionIsNull` | کاربر در بازه زمانی تعیین‌شده پاسخی ارسال نکرده است (منقضی شدن نشست). |
| **5** | `InvalidParameters` | پارامترهای ارسالی پذیرنده نامعتبر است. |
| **8** | `MerchantIpAddressIsInvalid` | آدرس سرور پذیرنده نامعتبر است (عدم تطابق IP پذیرنده با سیستم درگاه). |
| **10** | `TokenNotFound` | توکن ارسال شده یافت نشد. |
| **11** | `TokenRequired` | با این شماره ترمینال فقط تراکنش‌های توکنی قابل پرداخت هستند. |
| **12** | `TerminalNotFound` | شماره ترمینال ارسال شده یافت نشد یا غیرفعال است. |
| **21** | `MultisettlePolicyErrors` | محدودیت‌های مدل چند حسابی رعایت نشده است. |
