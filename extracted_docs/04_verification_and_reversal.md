# بخش ۴: تایید و برگشت تراکنش (صفحات ۱۵ تا ۱۹)

این بخش به روش‌های تایید نهایی تراکنش (Verify)، برگشت تراکنش (Reverse)، جلوگیری از دوبار مصرف شدن رسید دیجیتالی (Double Spending) و معرفی وب‌سرویس‌های متناظر می‌پردازد.

---

## ۴.۱. مکانیزم تایید تراکنش (Verify Transaction) - الزامی

پس از هدایت خریدار به آدرس بازگشت (RedirectURL) و دریافت پارامترهای Callback، پذیرنده باید برای اطمینان از صحت پرداخت، وب‌سرویس تایید تراکنش را فراخوانی کند.

> [!IMPORTANT]
> **محدودیت زمانی ۳۰ دقیقه‌ای:** پذیرنده پس از بررسی و صحت‌سنجی پارامترهای رسید تراکنش، حداکثر **۳۰ دقیقه** فرصت دارد تا تراکنش را با فراخوانی سرویس تایید، قطعی کند. در غیر این صورت، تراکنش به صورت خودکار برگشت خورده (Reverse) و مبلغ به حساب صاحب کارت بازمی‌گردد.

### الف) مقابله با پرداخت دوبار مصرف (Double Spending)
پذیرنده موظف است رسید دیجیتالی (`RefNum`) را در پایگاه داده خود جستجو کند تا مطمئن شود این رسید قبلاً مصرف نشده باشد. 
* رسید دیجیتالی به صورت کاملاً منحصر به فرد (Unique) از طرف پرداخت الکترونیک سامان صادر می‌شود.
* پرداخت الکترونیک سامان در برابر دو بار مصرف شدن رسید دیجیتالی هیچ واکنشی نشان نمی‌دهد؛ به این معنی که اگر یک رسید دیجیتالِ تایید شده دوباره ارسال شود، پرداخت الکترونیک سامان مجدداً آن را تایید می‌کند.
* پذیرنده باید تمهیدات لازم را در سمت خود پیاده‌سازی کند تا برای دو خرید متفاوت، از یک رسید دیجیتالی واحد استفاده نشود. این کار با ذخیره رسید دیجیتالی در بانک اطلاعاتی فروشگاه و مقایسه رسید جدید با رسیدهای قبلی انجام‌پذیر است.

### ب) بررسی همخوانی مبلغ (Amount Validation)
پذیرنده باید پس از فراخوانی وب‌سرویس تایید، پاسخ خروجی را بررسی کرده و مقدار بازگشتی خرید (`Amount`) را با مبلغ فاکتور مقایسه نماید. سه حالت ممکن است به وجود آید:
* **حالت A:** اگر خروجی تابع مثبت و برابر با مبلغ درخواستی فروشنده باشد، فروشنده می‌تواند کالا یا خدمات خود را ارائه کند (تراکنش موفق کامل).
* **حالت B:** اگر مبالغ برابر نباشند، کل مبلغ باید به حساب مشتری بازگردانده شود (تراکنش نامعتبر).
* **حالت C:** خروجی این تابع اگر منفی باشد، بیانگر رویداد خطایی است که پذیرنده باید کالا را تحویل ندهد.

## ۴.۲. وب‌سرویس تایید تراکنش (VerifyTransaction API)

* **آدرس وب‌سرویس:** `POST https://sep.shaparak.ir/verifyTxnRandomSessionkey/ipg/VerifyTransaction`
* **هدر درخواست:** `Content-Type: application/json`

### پارامترهای ارسالی درخواست (Verify Request Payload)

| نام پارامتر | نوع داده | توضیحات |
| :--- | :---: | :--- |
| `RefNum` | String | رسید دیجیتالی تراکنش (دریافتی در Callback). |
| `TerminalNumber` | Int64 | شماره ترمینال پذیرنده. **(توجه: نام فیلد در درخواست تایید TerminalNumber است و نه TerminalId)** |

### پارامترهای دریافتی پاسخ (Verify Response Payload)

| نام پارامتر | نوع داده | توضیحات |
| :--- | :---: | :--- |
| `Success` | Boolean | موفقیت یا ناموفق بودن کل درخواست وب‌سرویس تایید. |
| `ResultCode` | Int32 | کد وضعیت پاسخ عملیات تایید (مقدار `0` به معنای موفقیت است). |
| `ResultDescription` | String | شرح پاسخ خطا یا موفقیت به فارسی. |
| `TransactionDetail` | Object | شیء شامل جزئیات کامل تراکنش تایید شده (ساختار آن در ادامه آمده است). |

#### فیلدهای داخل شیء `TransactionDetail` (نوع داده `VerifyInfo`)

> [!CAUTION]
> **غلط املایی در فیلدهای خروجی (Typos in API Keys):**
> فیلدهای دریافتی در وب‌سرویس دارای غلط‌های املایی ثابت هستند. جهت دیسریالایز کردن بدون خطای پارسر در سیستم خود، از املاهای دقیق زیر استفاده کنید:
> * `OrginalAmount` (بدون حرف `i` بعد از `g`)
> * `StraceDate` (دارای حرف `S` در ابتدا)
> * `StraceNo` (دارای حرف `S` در ابتدا)

| نام فیلد در JSON | نوع داده | توضیحات |
| :--- | :---: | :--- |
| `RRN` | String | شماره مرجع تراکنش. |
| `RefNum` | String | رسید دیجیتالی تراکنش. |
| `MaskedPan` | String | شماره کارت خریدار به صورت ماسک شده (ماسک شده با ستاره). |
| `HashedPan` | String | شماره کارت خریدار به صورت هش شده (SHA256). |
| `TerminalNumber` | Int32 | شماره ترمینال پذیرنده. |
| `OrginalAmount` | Int64 | مبلغ ارسال شده به درگاه (مبلغ تراکنش به ریال). |
| `AffectiveAmount` | Int64 | مبلغ کسر شده واقعی از حساب صاحب کارت (ریال). |
| `StraceDate` | String | تاریخ و زمان انجام تراکنش. |
| `StraceNo` | String | شماره رهگیری تراکنش (کد رهگیری). |

#### نمونه درخواست تایید تراکنش (JSON)
```json
{
  "RefNum": "jJnBmy/IojtTemplUH5ke9ULCGtDtb",
  "TerminalNumber": 2015
}
```

> [!WARNING]
> **خطاهای مهم در پیاده‌سازی تایید تراکنش (Verify Errors):**
> 
> **۱. خطای 03:** در صورتی که ساختار ارسال درخواست تایید اشتباه یا ناقص باشد (مثلاً جابجایی پارامترها، خالی بودن یا اشتباه نوشتن نام فیلدهای `RefNum` یا `TerminalNumber`)، با خطای **`03`** مواجه خواهید شد.
> 
> **۲. خطای ۱۱۱ (منقضی شدن زمان وریفای):** در صورتی که پس از تراکنش موفق توسط کاربر، تابع وریفای ظرف مدت حداکثر ۳۰ دقیقه از سرور پذیرنده ارسال نگردد، تراکنش منقضی شده و درگاه خطای **`111`** (یا پیغام «امکان تایید تراکنش وجود ندارد») برگشت می‌دهد و پول به کارت خریدار بازمی‌گردد.
> 
> **۳. قالب کاراکترهای شبا (تسهیم چند حسابی):** در صورت استفاده از قابلیت تسهیم (چند حسابی)، پارامتر شبای ارسالی باید دقیقاً مطابق با شباهای تایید شده روی ترمینال باشد و عبارت ابتدایی شبا یعنی **`IR`** حتماً باید با **حروف بزرگ انگلیسی** درج گردد (در صورت ارسال با حروف کوچک `ir` تراکنش ناموفق خواهد بود).

#### نمونه پاسخ موفق تایید تراکنش (JSON)

    "RRN": "14226761817",
    "RefNum": "50",
    "MaskedPan": "621986****8080",
    "HashedPan": "b96a14400c3a59249e87c300ecc06e5920327e70220213b5bbb7d7b2410f7e0d",
    "TerminalNumber": 2001,

## ۴.۳. اصلاحیه تراکنش یا برگشت تراکنش (Reverse Transaction)

در صورتی که پذیرنده بخواهد تراکنش تایید شده را لغو کند و مبلغ را به حساب مشتری برگشت بزند، می‌تواند تا حداکثر **۵۰ دقیقه** پس از انجام تراکنش، سرویس برگشت را فراخوانی کند.

* **آدرس وب‌سرویس:** `POST https://sep.shaparak.ir/verifyTxnRandomSessionkey/ipg/ReverseTransaction`
* **هدر درخواست:** `Content-Type: application/json`

### درخواست و پاسخ برگشت (Reverse JSON)
ساختار فیلدهای ارسالی و دریافتی این وب‌سرویس کاملاً منطبق بر وب‌سرویس تایید تراکنش (`VerifyTransaction`) است و در صورت موفقیت، مقدار `ResultCode` برابر `0` خواهد بود.

---

## ۴.۴. جدول کدهای پاسخ وب‌سرویس‌های تایید و برگشت (Result Codes)

کدهای خطای بازگشتی در فیلد `ResultCode` درگاه پرداخت در وب‌سرویس‌های تایید (`Verify`) و اصلاحیه (`Reverse`) عبارتند از:

| کد پاسخ (`ResultCode`) | متدهای مرتبط | شرح کد خطا |
| :---: | :---: | :--- |
| **0** | `verify` \| `reverse` | عملیات با موفقیت انجام شد (موفق). |
| **2** | `verify` \| `reverse` | درخواست تکراری می‌باشد (این تراکنش قبلاً با موفقیت تایید یا برگشت خورده است). |
| **-2** | `verify` | تراکنش یافت نشد. |
| **-6** | `verify` | بیش از نیم ساعت از زمان اجرای تراکنش گذشته است و تراکنش منقضی شده است. |
| **-104** | `verify` \| `reverse` | ترمینال ارسالی غیرفعال می‌باشد. |
| **-105** | `verify` \| `reverse` | ترمینال ارسالی در سیستم موجود نمی‌باشد. |
| **-106** | `verify` \| `reverse` | آدرس آی‌پی درخواستی غیرمجاز می‌باشد (آی‌پی سرور پذیرنده با آی‌پی ثبت‌شده مطابقت ندارد). |
| **5** | `verify` | تراکنش برگشت خورده می‌باشد (قبلاً برگشت خورده است). |
